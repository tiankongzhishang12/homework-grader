# 软件项目基础实训评分系统前端页面设计总结

## 1. 文档目的

本文用于整理 `software-project-practicum` 场景下的前端页面设计思路，作为后续页面细化、Vue 实现和接口对接的基础说明。

当前前端方案默认服务于以下目录下的专项评分流程：

`D:\grade code\homework-grader\software-project-practicum`

当前固定工作区为：

`software-project-practicum/workspace/practicum-batch`

---

## 2. 前端定位

这个前端系统的定位不是普通的文件上传页，而是一个面向教师使用的“评分评审工作台”。

它需要完成三类核心任务：

- 上传并发起评分任务
- 查看整批学生作业的整体评分情况
- 查看单个学生的详细评分依据与问题定位

因此，前端页面设计必须围绕以下三条主线展开：

- 任务流
- 结果流
- 问题定位流

---

## 3. 整体页面结构

当前建议前端页面采用以下结构：

1. 总览页 `Dashboard`
2. 评分标准页 `Rubric Management`
3. 学生列表页 `Student List`
4. 学生详情页 `Student Detail`
5. 统计分析页 `Analytics`

在实际使用过程中，还需要在页面顶部或总览页中提供批次操作入口，例如：

- 上传作业
- 开始批改
- 导出报表
- 刷新结果

---

## 4. 页面设计目标

### 4.1 总览页

总览页的目标是帮助教师快速了解当前批次的整体情况。

主要回答以下问题：

- 当前这批作业处理到什么阶段了
- 班级整体分数如何
- 哪些学生需要优先查看
- 哪类问题最常见

建议包含以下模块：

- 批次信息卡
- 任务进度卡
- 核心统计卡
- 风险学生列表
- 分数分布图
- 高频问题排行

### 4.2 学生列表页

学生列表页的目标是帮助教师快速筛选出高风险学生，并进入详情页查看。

主要回答以下问题：

- 哪些学生分数低
- 哪些学生待复核
- 哪些学生置信度低
- 哪些学生存在未覆盖需求或一致性问题

建议包含以下模块：

- 搜索栏
- 条件筛选栏
- 快捷筛选标签
- 学生评分表格

说明：

- 快捷筛选标签用于在同一张表格中快速过滤学生
- 当前第一轮原型中，列表顶部“已应用筛选条件”标签已移除，页面只保留更直观的搜索与快捷筛选

### 4.3 评分标准页

评分标准页用于支持不同教师按各自要求组织和复用评分标准。

主要回答以下问题：

- 当前有哪些评分标准可用
- 哪些是我的评分标准
- 哪些是共享或课程组评分标准
- 当前批次正在使用哪一份评分标准
- 如何新增、复制和编辑评分标准

建议包含以下模块：

- 评分标准列表区
- 当前选中评分标准工作区
- 评分维度预览
- 门禁规则预览
- 评语规则预览
- 校验与预览区

状态建议固定为：

- 编辑中
- 已完成
- 使用中

### 4.4 学生详情页

学生详情页是整个前端系统的核心页面。

该页面需要解决的核心问题不是“学生得了多少分”，而是：

- 为什么得这个分
- 分数依据是什么
- 哪些需求没有被承接
- 哪些设计存在一致性问题
- 哪些地方需要教师重点复核

建议包含以下模块：

- 顶部总评卡
- 维度评分概览
- 单维度详情
- Traceability 追踪分析
- 一致性问题面板
- 门禁检查
- 原始材料摘要

### 4.5 统计分析页

统计分析页用于从班级整体角度分析评分结果。

主要回答以下问题：

- 分数分布是否合理
- 哪些维度普遍得分偏低
- 高频扣分问题是什么
- 哪些未覆盖需求最常见
- 低置信度样本多不多

建议包含以下模块：

- 分数分布图
- 等级分布图
- 各维度平均分图
- 高频扣分维度排行
- 高频未覆盖需求排行
- 低置信度分布图

## 5. 推荐页面主流程

从教师的实际使用路径出发，推荐前端主流程如下：

1. 上传学生作业
2. 配置或选择评分标准
3. 开始批改
4. 在总览页查看整体情况
5. 在学生列表页筛选高风险学生
6. 在学生详情页查看具体问题
7. 在统计分析页查看班级共性问题
8. 导出评分报表

这个流程意味着前端不是单一页面，而是一个有明确操作顺序的工作台。

---

## 6. 当前页面设计重点

结合现有评分 JSON 结构，页面设计重点应放在以下几个方面。

### 6.1 风险优先

界面不应只是机械展示所有学生，而应优先让教师看到：

- 待复核学生
- 低置信度学生
- 未覆盖需求较多的学生
- 一致性问题较多的学生

### 6.2 证据优先

界面不应停留在总分和等级上，而应突出展示：

- 评分证据 `evidence`
- 评分推理 `reasoning`
- 命中点 `matched_core_points`
- 缺失点 `missing_points`
- 改进建议 `improvement`

### 6.3 追踪优先

该项目的核心特色是“需求 -> 概设 -> 详设”的追踪式评分，因此应将追踪关系作为详情页重点内容进行展示。

建议在详情页中显式展示：

- 抽取需求
- HLD 覆盖情况
- LLD 覆盖情况
- 未覆盖需求

### 6.4 问题优先

页面需要支持快速呈现关键问题，例如：

- 某个需求在后续设计中没有承接
- 某个章节存在模板占位残留
- 前后文设计对象不一致

例如“薪资结算进度跟踪未覆盖”这类问题，应在详情页和风险摘要中被明显高亮，而不是埋在原始 JSON 中。

## 7. 页面功能拆分建议

### 7.1 总览页功能

建议包括：

- 显示批次名称
- 显示最近更新时间
- 显示批次级操作入口：上传作业、开始批改、导出 Excel、刷新结果
- 显示学生总数、已完成数、待处理数
- 显示平均分、待复核人数、低置信度人数
- 显示未覆盖需求总数、一致性问题总数
- 展示风险学生列表
- 展示高频问题和高频扣分维度

### 7.2 评分标准页功能

建议包括：

- 显示评分标准列表
- 区分我的评分标准与共享评分标准
- 对我的评分标准支持查看、编辑、复制
- 对共享评分标准支持查看、复制
- 支持新增评分标准
- 展示评分维度、门禁规则、评语规则和校验预览
- 展示状态：编辑中、已完成、使用中

### 7.3 学生列表页功能

建议包括：

- 按学号、姓名、匿名 ID 搜索
- 使用快捷筛选标签在同一张列表中切换：
  - 全部
  - 待复核
  - 低置信度
  - 未覆盖需求
  - 一致性问题
- 表格排序
- 点击进入详情页

### 7.4 学生详情页功能

建议包括：

- 展示学生基本信息
- 展示总分、等级、置信度、门禁状态
- 展示一句话结论
- 展示所有评分维度卡片
- 查看单维度的证据、推理、命中点、缺失点、建议
- 展示追踪分析
- 展示一致性问题
- 展示门禁检查
- 展示原始文档摘要

### 7.5 统计分析页功能

建议包括：

- 分数分布分析
- 等级分布分析
- 各评分维度平均分分析
- 高频问题排行
- 高频未覆盖需求排行
- 低置信度分析

## 8. 第一版推荐范围

为了尽快形成可讨论、可演示的页面原型，建议第一版前端先只实现以下内容：

- `Dashboard`
- `Rubric Management`
- `Student List`
- `Student Detail`

统计分析页可以放到第二阶段补充。

同时，第一版建议采用以下边界：

- 只读模式
- 单批次模式
- 使用本地 mock 数据
- 不接后端
- 不实现登录

这样可以先把页面逻辑和信息组织方式确定下来，再进入开发和联调阶段。

---

## 9. 界面风格建议

建议整体采用“克制、专业、偏评审”的视觉风格。

具体建议如下：

- 左侧固定导航
- 顶部展示批次信息和刷新入口
- 主区域采用卡片式布局
- 强调风险色而不是装饰色
- 绿色表示通过
- 橙色表示待关注
- 红色表示严重问题
- 主基调使用蓝灰色或中性色

不建议将页面做成过于花哨的教学系统风格，而应更贴近“教师评审工作台”。

---

## 10. 当前总结

当前 `software-project-practicum` 前端页面设计可以总结为：

- 这是一个围绕专项评分流程构建的教师评审工作台
- 页面分为总览、学生列表、学生详情、统计分析四部分
- 页面主流程为：上传 -> 预处理 -> 评分 -> 查看总览 -> 筛选学生 -> 查看详情 -> 导出
- 页面设计重点不是单纯展示分数，而是解释分数、定位问题、体现追踪关系

这份设计总结将作为后续页面功能说明、Vue 页面实现和接口设计的基础文档。
