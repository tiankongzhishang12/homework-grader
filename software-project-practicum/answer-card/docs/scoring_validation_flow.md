# 评分标准验证流程

## 1. 目标

本流程用于验证当前答题卡自动评分方案是否达到可用水平，并定位评分误差来自哪里。

验证对象包括：

- 客观题标准答案 `standard_answers.yaml`
- 填空题候选答案归并结果
- 主观题评分标准 `inferred_subjective_rubric.yaml` / 后续正式 `scoring_rubric.yaml`
- 整体自动评分流程中的提示词、阈值和人工复核策略


## 2. 当前数据划分

当前教师已批改扫描件共 `206` 份，按如下方式使用：

- `100` 份用于生成评分标准  
  作用：归纳客观题标准答案、填空题候选答案、主观题 rubric
- `30` 份用于验证  
  作用：完全不参与评分标准归纳，只用于验证自动评分效果
- `76` 份用于缓冲  
  作用：后续补样、异常排查、追加调优、最终封闭测试候选池

注意：

- 验证集 `30` 份不得参与评分标准归纳
- 缓冲集 `76` 份默认不参与当前轮验证
- 若后续重做评分标准，需要重新固定训练/验证边界


## 3. 验证原则

- 只用教师已批改试卷作为真值来源
- 验证时不得让系统读取教师给分区域
- 自动评分结果必须逐题与教师真实分对比，不能只比较总分
- 误差分析必须区分“评分标准错误”和“识别/理解错误”
- 任何调优动作完成后，都必须重新在同一验证集上复测


## 4. 验证输入与输出

### 输入

- 验证集试卷 ID 列表
- 验证集原始扫描件
- 当前版本评分标准
- 当前版本自动评分脚本/流程

### 输出

- 每份试卷自动评分结果
- 每份试卷教师真实分
- 自动分与教师分逐题对比表
- 汇总误差报告
- 典型错例清单


## 5. 用 100 份样本重建评分标准的方案

当前最合理的做法，不是把 100 份样本简单重跑一次，而是把这 100 份当成正式的评分标准构建集，按阶段完成。

### 阶段 1：把 100 份样本整理成可用真值集

目标：

- 先把数据底座打稳，再进入评分标准归纳

要做的事：

- 跑完这 `100` 份教师样本的第一阶段解析
- 拿到每份卷子的教师总分、客观题分区分数、`26-33` 各题分数
- 标记异常样本：
  - 总分与分项和不一致
  - 某题教师分不清晰
  - 图像模糊或裁剪异常
- 生成一张样本画像表，至少包含：
  - `paper_id`
  - 总分段
  - `26-33` 各题分数段
  - 是否异常
  - 是否属于图示题低分样本

样本最好进一步区分为：

- 可直接用于评分标准归纳
- 只可用于部分题型归纳
- 需人工复核样本

### 阶段 2：分别重建三类评分标准

不要把所有题型混在一起处理，应分三条线更新。

#### 2.1 客观题

- 用 100 份样本重新跑一次客观题答案反推
- 统计每题一致率
- 对冲突题做人工确认
- 输出新的 `standard_answers.yaml`

#### 2.2 填空题

- 用 100 份样本重新做候选答案聚合
- 形成：
  - canonical answer
  - aliases
  - review_required 条件
- 对高冲突空进行人工确认
- 更新填空题评分规则

#### 2.3 主观题

- 用 100 份样本重新做主观题样本抽取
- 每题按分档抽样：
  - 满分
  - 高分
  - 中分
  - 低分
  - 零分
- 重新归纳 `26-33` 的 rubric
- 重点关注：
  - `26`
  - `30`
  - `31`
  - `32`
  - `33`

这些题最容易出现得分点不稳或图示题误判。

### 阶段 3：产出候选版和差异分析版

不要直接产出最终标准，而是同时生成：

- 候选正式版 `scoring_rubric.yaml`
- 差异分析报告

差异分析至少应包含：

- 新旧 rubric 哪些题变化了
- 每题新增或删减了哪些得分点
- 哪些题的分值分配发生调整
- 哪些题仍然保留 `teacher_review_required`

### 阶段 4：立刻在 30 份验证集上闭环验证

新的评分标准生成后，必须立刻做独立验证，不能凭感觉判断“标准变好了”。

验证时要做：

- 用新的评分标准跑 `30` 份独立验证集
- 逐题对比教师分
- 统计：
  - 总分 MAE
  - 各题 MAE
  - 完全一致率
  - 容差命中率
  - 各题系统性偏高/偏低
- 如果某题偏差显著，再回到阶段 2 只修该题标准，不做全局乱改


## 6. 验证步骤

### 第一步：固定验证集

使用 `30` 份验证集样本，名单固定，不允许在验证过程中随意替换。

建议保存：

- `validation_ids.txt`
- 对应原图路径清单

### 第二步：准备真值

从教师已批改试卷中提取教师真实分，形成验证真值表。

真值至少包含：

- `paper_id`
- `total_score`
- 客观题分区分数
- `26-33` 各题教师分数

注意：

- 若存在“总分与分项和不一致”，应额外打标
- 若某题教师分不清晰，应进入人工复核列表

### 第三步：执行盲测评分

对验证集运行自动评分流程，但评分时只能看学生答题内容，不能读取教师批改区。

建议流程：

- 客观题：按当前标准答案自动判分
- 填空题：按当前 canonical + aliases 判分
- 主观题：按当前 rubric 自动评分
- 输出每题得分、证据、置信度、是否建议人工复核

### 第四步：生成逐题对比表

将自动评分结果与教师真实分合并，形成逐题误差表。

建议字段：

- `paper_id`
- `teacher_total`
- `auto_total`
- `total_diff`
- `teacher_q26` ~ `teacher_q33`
- `auto_q26` ~ `auto_q33`
- `diff_q26` ~ `diff_q33`
- `needs_review`
- `confidence`

### 第五步：统计核心指标

至少统计以下指标：

- 总分平均绝对误差 `MAE`
- 各题平均绝对误差
- 总分完全一致率
- 各题完全一致率
- 总分容差命中率：误差 `<= 0.5`、`<= 1.0`
- 各题容差命中率
- 各题平均偏差方向  
  正值偏高，负值偏低
- 人工复核触发率

### 第六步：做错例分析

每题至少挑选以下样本复盘：

- 误差最大的 3 份
- 自动高分但教师低分的样本
- 自动低分但教师高分的样本
- 低置信度样本

复盘时判断误差来源属于哪类：

- 标准答案不准
- rubric 得分点不合理
- 提示词导致模型过度脑补
- 图示题识别错误
- 填空别名覆盖不足
- 教师真值本身存在歧义


## 7. 验证通过标准

建议把验证结果分成 3 档判断：

### 可直接进入试运行

- 总分平均绝对误差较低
- `26-33` 大部分题目误差稳定
- 高误差样本比例可控
- 人工复核量在可接受范围内

### 可继续调优

- 总分误差中等
- 某几道题误差明显偏高
- 主观题存在系统性偏高或偏低
- 错误模式清晰，可继续改 rubric 或 prompt

### 不建议上线

- 总分误差较大
- 多道主观题偏差严重
- 图示题评分不稳定
- 自动评分与教师分经常相差超过 2 分


## 8. 调优闭环

验证不是一次性动作，而是闭环：

1. 固定评分标准版本
2. 在 30 份验证集上跑盲测
3. 统计误差与错例
4. 只修改一类问题  
   例如 rubric、prompt、阈值、别名表
5. 在同一验证集上重新复测
6. 比较版本前后指标变化

注意：

- 每次尽量只改一类因素，避免无法定位提升来源
- 如果验证集反复被“看穿”，后续应从 `76` 份缓冲集中抽新验证集做二次验证


## 9. 当前阶段建议

结合当前已完成工作，建议按下面顺序推进：

1. 用 `100` 份样本重新生成更稳的评分标准
2. 固定 `30` 份验证集真值
3. 实现一次完整盲测评分
4. 生成逐题误差报告
5. 先优先修正误差最大的题  
   重点通常是 `26`、`30`、`31`、`32`、`33`
6. 当 30 份验证结果稳定后，再从 `76` 份缓冲集中抽一部分做第二轮验证

## 11. 一句话结论

当前最合理的验证方式是：

先用 `100` 份样本生成评分标准，再在独立的 `30` 份验证集上做盲测回放，通过逐题误差分析判断评分标准是否可靠，最后把 `76` 份缓冲样本留给后续补样和二次验证。
